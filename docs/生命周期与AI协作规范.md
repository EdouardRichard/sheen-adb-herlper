# Sheen ADB 助手：生命周期与 AI 协作规范

> 状态：已确认（2026-07-11）  
> 适用范围：所有版本、全部源码、文档、依赖、构建与发布流程。  
> 目的：让人类和 AI 在多次迭代中以同一套边界演进项目，避免协议逻辑、UI 状态、权限与依赖失控。

## 1. 规范优先级

发生冲突时按以下顺序决策：

1. 法律、平台审核规则、用户明确指令和安全要求。
2. 本文的不可变约束。
3. 当前版本需求文档（例如 [`v0.01-需求文档.md`](v0.01-需求文档.md)）。
4. ADR（架构决策记录）、接口契约和测试。
5. 实现便利性或个人偏好。

需求未明确时，优先选择更小的权限、更小的依赖面、更少的持久化数据和可逆的实现；不得擅自扩大功能范围。

## 2. 不可变约束

- 产品保持纯本地、无账号、无后端、无广告、无统计/遥测 SDK；v0.01 不含任何付费机制。
- 主控端 `minSdk 30`，基线为 HyperOS 3 / Android 16，`compileSdk` 与 `targetSdk` 固定为 36，除非以 ADR 批准变更。
- 无 Root 依赖；不得绕过 ADB 授权、无线调试、系统确认或使用无障碍服务自动开启调试。
- 配对码、ADB 主机私钥、设备日志、设备标识、Shell 输出均视为敏感数据，默认不落盘、不上传、不写日志。
- 同一时刻只允许一个活跃 ADB Session。
- 首版不做局域网扫描、后台 Logcat、后台常驻、自启动、所有文件访问或相机扫码。
- 开源仓库使用 **Apache-2.0**；商店包与开源构建在功能上必须等价。

## 3. 技术基线

| 类别 | 强制选择 |
|---|---|
| 语言 | Kotlin；新业务代码禁止 Java |
| UI | Jetpack Compose + Material 3 |
| 异步 | Kotlin Coroutines + Flow；禁止回调链与 `GlobalScope` |
| 状态 | 单向数据流：UI Event → ViewModel → Use Case/Repository → immutable UI State |
| 本地存储 | DataStore 保存非敏感偏好与设备档案元数据；密钥使用 Android Keystore 保护 |
| 文件导出 | Storage Access Framework；禁止以便利为由申请所有文件访问 |
| 导航 | 单一导航宿主；目的地只传稳定 ID，不传 Socket、ViewModel 或大对象 |
| 构建 | Gradle Kotlin DSL、Version Catalog、固定并审查依赖版本 |

## 4. 模块边界

项目必须保持以下逻辑模块；小型实现可暂在同一物理模块中保留同名包，但在首次稳定迭代前拆分为对应 Gradle 模块。新增功能默认创建独立 `feature` 模块。

```text
:app                 应用装配、导航、依赖注入；唯一可组合 feature 的位置
:core:adb            ADB 协议、TLS/配对、连接状态、主机身份、命令与流
:core:data           DataStore、设备档案、导出接口
:core:ui             主题、设计令牌、通用无状态组件
:feature:devices     首页、连接/配对、设备列表
:feature:overview    设备概览
:feature:shell       终端
:feature:processes   进程监控
:feature:logcat      Logcat
```

依赖方向只能从 `app/feature → core`。Feature 之间不得直接依赖；需要共享的稳定模型、接口或 UI 组件必须下沉到相应 `core`。`core:adb` 不得依赖任何 Feature、Compose 页面或 Android 导航类型。

## 5. ADB 协议与会话规则

### 5.1 协议适配层

- 第三方 ADB 库只能位于 `core:adb` 的实现层；对外暴露项目自有的接口、值对象与错误模型。
- 禁止将第三方库的连接、异常、密钥或回调类型泄漏至 Feature 或 UI。
- 选用库前必须完成 PoC：Android 11+ 配对、HyperOS 3/Android 16 配对、本机 `127.0.0.1`、旧式 `:5555`（如有设备）以及主动取消。
- 禁止直接复制许可证不兼容的项目代码。所有借鉴的源码、协议资料和许可证必须记录来源。

### 5.2 会话所有权

- `AdbSessionManager` 是唯一有权创建、持有和关闭 Socket/TLS 会话的对象。
- UI、ViewModel、Repository 和 Feature 禁止直接建立 Socket 或执行原始 ADB 包。
- 每个操作必须绑定活跃会话 ID；会话切换、断开或失效后，旧操作的结果不得写入新设备 UI。
- 每一个连接、配对、Shell、进程刷新与 Logcat 流都必须有超时、取消、结构化错误和 `finally` 清理。
- 连接状态只能由会话管理器发布，UI 不得自行推断“已连接”。

### 5.3 命令契约

- ADB 命令集中定义在 `core:adb` 的命令/能力层，禁止在 Composable、ViewModel 或 Feature 中拼接 Shell 字符串。
- 对 ROM 差异采用“能力探测 → 兼容实现 → 明确降级”策略；不得依赖单一厂商的输出格式。
- 原始 Shell 终端是唯一允许用户输入任意命令的通道；用户原文必须原样执行，不得静默过滤或改写。
- 业务页面只能调用命名的高层能力，例如 `loadDeviceInfo()`、`listProcesses()`、`streamLogcat()`。

## 6. UI 与状态纪律

- Composable 必须无副作用或将副作用放进受控 Effect；不得直接访问 DataStore、文件、Socket、ADB 客户端或启动协程执行业务。
- ViewModel 是页面状态唯一来源，公开不可变 `StateFlow<UiState>` 与明确的事件入口。
- 所有页面都必须表达：加载、内容为空、可恢复错误、不可恢复错误、离线/未连接、取消中（如适用）。
- UI 使用设计令牌，不散落硬编码颜色、字体、间距和文本；用户可见字符串集中管理。
- 不因实现容易而引入隐藏手势、全局可变单例状态、重复导航或跨页面直接修改状态。

## 7. 数据、隐私与日志

### 7.1 数据分级

| 分级 | 示例 | 规则 |
|---|---|---|
| 高敏感 | ADB 私钥、证书、配对码 | 仅内存或 Keystore；绝不打印、导出、提交或上传 |
| 敏感 | Logcat、Shell 输出、设备序列号、IP、构建指纹 | 默认仅内存；用户明确复制/导出时才离开；测试使用脱敏样本 |
| 本地元数据 | 显示名、主机、端口、最近连接时间、主题设置 | 最小化保存在 DataStore；用户可一键清除 |
| 公开数据 | 版本号、开源许可证 | 可随应用发布 |

### 7.2 强制规则

- 日志系统默认脱敏 IP、序列号、路径和命令输出；Debug 构建如需完整诊断，必须由显式本地开关启用且不得进入 Release。
- 不接入崩溃上报、分析、推送、广告、远程配置或 WebView 远程内容。
- 导出只使用用户选择的 SAF 目标；不缓存导出副本。
- “清除所有本地数据”须删除档案、密钥引用、偏好和临时文件，并有自动化/手工验证。

## 8. 权限与商店合规

- 每一个 Manifest 权限必须在 `docs/权限矩阵.md` 中登记：功能、触发时机、拒绝后的降级、审核说明和测试证据。
- 未登记权限不得合入。新危险权限、特殊权限或后台能力须先经 ADR 批准。
- 禁止 `MANAGE_EXTERNAL_STORAGE`、无关的定位/相机/通讯录/电话/短信权限、开机广播和常驻前台服务，除非未来版本的需求文档和 ADR 同时批准。
- 不得申请与功能无关的包可见性；业务不得以读取主控端应用列表为手段实现被控端能力。
- 发布前按小米审核规范核对功能说明、权限说明、隐私政策、截图、测试路径和测试设备。小米要求权限的用途与范围符合最小必要原则，并在审核时说明用途；参见 [权限使用及申请标准](https://dev.mi.com/xiaomihyperos/documentation/detail?pId=2156)。

## 9. 依赖与供应链

- 许可白名单：Apache-2.0、MIT、BSD-2-Clause、BSD-3-Clause。其他许可证必须做 ADR 兼容性审查。
- 禁止引入广告、统计、支付、云端、动态代码加载、未经审查的二进制 AAR，以及功能重复的大型框架。
- 每个新增依赖必须在 PR/变更说明中写清：用途、许可证、版本、体积/方法数影响、维护状态、替代方案和移除计划（如适用）。
- 依赖版本只允许通过 Version Catalog 修改；定期执行漏洞检查和许可清单生成。
- 所有密钥、签名文件、`local.properties`、导出日志和真实设备数据必须由 `.gitignore` 覆盖并在 CI 中扫描。

## 10. 测试与质量门禁

### 10.1 每次变更必须满足

1. 编译通过、格式化和静态检查通过。
2. 新增/修改纯逻辑时包含单元测试；至少覆盖正常、边界、取消和错误路径。
3. 修改 UI 状态时覆盖加载、成功、空、错误、断开/取消的关键状态测试。
4. 修改 ADB 层时覆盖会话归属、超时、取消、资源关闭和错误映射。
5. 文档、版本记录和权限矩阵与代码同步更新。

### 10.2 发布前必须满足

- 测试矩阵覆盖 Android 11、HyperOS 3/Android 16 和至少一台非小米设备（可获得时）；覆盖本机 `127.0.0.1`、远端配对、重连、掉线和旧式 `:5555`（可获得时）。
- 真机验证设备概览、Shell 取消、进程降级、Logcat 开始/停止/导出和清除全部本地数据。
- Release 构建开启 R8/资源收缩（在验证兼容性后），无 Debug 日志、测试后门、明文密钥或开发服务器地址。
- 生成第三方许可证清单、隐私政策和小米审核演示材料。

## 11. 变更、版本与 ADR

- 采用语义化版本：`MAJOR.MINOR.PATCH`；v0.x 期间仍需记录破坏性变更。
- 功能需求通过 `docs/v0.xx-需求文档.md` 追加；已发布需求不直接改写历史结论，使用“修订记录”说明变化。
- 以下变更必须新增 ADR：SDK/语言/UI 框架、模块依赖方向、ADB 库、密钥存储、权限、网络边界、数据持久化、日志策略、开源许可证和商店策略。
- ADR 至少包含：背景、候选方案、决策、理由、影响、回滚方式、日期和负责人。
- 任何跨模块重构都必须单独提交并说明迁移计划；不得夹带在功能提交中。

## 12. AI 协作工作流

AI 在修改项目前必须依序执行：

1. 阅读本规范、当前版本需求、相关 ADR 和目标模块的现有代码/测试。
2. 用一句话说明本次改动对应的需求条目、影响的模块和不触碰的边界。
3. 先补/改接口、模型和测试，再实现；协议或权限不明确时先提出问题或创建 ADR，不凭猜测接入。
4. 只改与任务有关的文件；不得删除用户已有改动、格式化无关目录或引入顺手重构。
5. 运行适当验证并报告命令、结果、未覆盖风险和文档变更。

AI 严禁：

- 在 UI 层直接网络连接、运行 Shell、读写密钥或文件。
- 使用 `GlobalScope`、阻塞主线程、吞掉异常、无界缓存/Flow、全局可变连接对象。
- 记录敏感数据、添加遥测 SDK、扩大权限、自动修改系统调试开关。
- 将“设备不支持/ROM 差异”伪装为成功，或把未知错误误导为配对失败。
- 未经明确批准改变本规范中的不可变约束。

## 13. 发布与安全响应

- GitHub 公开仓库使用 Apache-2.0，并包含 `LICENSE`、`NOTICE`、`SECURITY.md`、隐私政策、贡献指南和第三方许可证说明。
- 生产签名密钥只能在受控本地或 CI 密钥库中使用，绝不进入源码、Issue、日志或发布包以外的附件。
- 安全问题通过私密渠道接收；确认后先止损、评估受影响版本、修复、回归、发布，再公开披露。
- 若发现私钥、配对材料、用户日志或签名密钥泄露，立即撤销/轮换受影响密钥、删除公开暴露、发布修复说明并复盘流程漏洞。
